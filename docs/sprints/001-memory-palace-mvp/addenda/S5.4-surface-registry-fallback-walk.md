## Surface registry implementation approach

The surface registry is a `Set<string>` constant (`WEB_SURFACES`) adjacent to the
dispatcher in `InscriptionLens.svelte`. It contains exactly the five sprint-001
surfaces: `scroll`, `tablet`, `book-spread`, `etched-wall`, `floating-glyph`.
This is a lens-local implementation detail per ADR 2026-04-24-surface-registry §2 —
not a wire-format concept, not a shared module. Each future engine (Unreal, Blender)
would define its own registry.

## Fallback algorithm

The walk is a simple linear scan with a `visited: Set<string>` for cycle detection:

```
resolveSurface(requested, fallbackChain):
  if requested ∈ WEB_SURFACES → return requested immediately (no log)
  visited = {requested}
  for candidate in fallbackChain[0..MAX_WALK_HOPS]:
    if candidate ∈ visited → emit surface-fallback-cycle; break
    visited.add(candidate)
    if candidate ∈ WEB_SURFACES → emit surface-fallback; return candidate
  emit surface-fallback(resolved: "scroll")
  return "scroll"
```

Walk terminates in O(min(|fallbackChain|, MAX_WALK_HOPS)) — bounded at 10 hops
as the DoS guard per ADR implementation note. `scroll` is always the terminal step.

`resolveSurface` is called synchronously in `onMount` before the async
`inscriptionBody` fetch, so the surface is resolved and the placeholder is shown
immediately while bytes are loading.

## CAS plumbing decision

No pre-existing `casRead` helper or CAS directory concept existed at the store level.
The inscription body path in prior sprints was: file-watcher reads from `sourcePath`
(a local filesystem path stored per-inscription in the watcher state, not in the DB).

For `inscriptionBody(fp)`, the approach taken:

**Server**: `ServerStore` gets an optional `casDir` constructor arg (also read from
`PALACE_CAS_DIR` env var). The verb queries `source_blake3` from the Inscription node,
reads `<casDir>/<source_blake3>` from disk, and hash-verifies before returning.
No raw filesystem path leaks to the caller.

**Browser**: fetches from same-origin `/cas/<source_blake3>` (SEC6: no cross-origin).
Hash-verifies the response.

This is surfaced as a **spec-gap** that implementors must configure: the store needs
`casDir` set to a directory where inscription body files are named by their Blake3 hash.
In practice the file-watcher already writes to palace directories; the CAS directory
is a separate flat content-addressed store that the palace-mint / inscribe bridges
should populate. This is tracked as a TODO for the palace-inscribe bridge (Epic 3).

## Floating-glyph glyph-atlas approach

Full per-glyph 3D mesh (Three.js `TextGeometry` + `FontLoader`) was deferred to
Growth tier. The MVP renders each glyph as an HTML `<span>` with CSS animation
(`glyph-drift` keyframes). This satisfies AC1 (text visible, `getByText` passes)
without the WASM font-atlas pipeline. The soft-warn >2 KB is implemented via
`onMount` + `TextEncoder` byte-count check.

Body is capped at 256 rendered glyphs (`GLYPH_RENDER_LIMIT`) to bound the DOM
node count. Beyond the limit, a `…` ellipsis replaces the remainder.

## Why HTML-only surfaces for MVP

The Threlte `<T.*>` components require being inside a `<Canvas>` context
(which establishes the Threlte scheduler via Svelte context). Surface components
rendered as children of a conditional outside `<Canvas>` would trigger
`useScheduler can only be used in a child component to <Canvas>`. Rather than
wire a Canvas + 3D font atlas for MVP, all five surfaces render as HTML overlays
with CSS styling that evokes the surface aesthetics (scroll parchment, stone tablet,
book pages, etched low-contrast, floating animated glyphs). The `<Canvas>` + mesh
path is preserved for Growth tier by the surface component structure.
