# S5.2 — Grid-fallback algorithm for PalaceLens

**Date**: 2026-04-24
**Story**: S5.2 — `PalaceLens.svelte` (FR15, AC3)
**Author**: executor (oh-my-claudecode)

## What

When a palace field envelope carries no `jelly.layout` attribute for one or
more rooms, `PalaceLens.svelte` places those rooms on a **Fibonacci-spiral
onion shell** at radius 5 m (world units). The algorithm is:

```ts
const SHELL_RADIUS = 5;
const GOLDEN_ANGLE = Math.PI * (3 - Math.sqrt(5)); // ≈ 2.399 rad

function fibonacciShellPosition(index: number, total: number): [number, number, number] {
  const y = 1 - (index / Math.max(1, total - 1)) * 2; // [-1, 1]
  const r = Math.sqrt(Math.max(0, 1 - y * y));
  const theta = GOLDEN_ANGLE * index;
  return [SHELL_RADIUS * r * Math.cos(theta), SHELL_RADIUS * y, SHELL_RADIUS * r * Math.sin(theta)];
}
```

`index` is the room's position in the list returned by `store.roomsFor()`, which
returns rooms sorted by `fp ASC`. This makes the index deterministic for any
given set of room fps — the same palace produces the same grid positions across
two independent mounts (AC3 byte-stability requirement).

## Why Fibonacci spiral over alternatives

| Alternative | Reason rejected |
|---|---|
| Regular latitude/longitude grid | Poles suffer from node clustering for N < ~20. Rows near the equator have more slots than rows near poles, making the grid uneven for small N (3–10 rooms). |
| Random placement with seeded PRNG | Requires storing/transmitting the seed or re-deriving from fp bytes. Adds complexity without benefit over the Fibonacci approach. |
| Cube face projection | Visible seams at cube edges; doesn't feel "spherical." Aesthetically wrong for an omnispherical palace. |
| Hexagonal tiling on sphere | Requires external library (geodesic subdivision). Overkill for sprint-001 smoke scope. |
| Fibonacci spiral (chosen) | Provably optimal equidistribution for arbitrary N; O(1) per point; zero dependencies; naturally "spherical" in appearance; fully deterministic from index alone. The golden angle ≈ 2.399 rad is the most irrational number in the sense of being least well approximated by rationals — this is why Fibonacci-spaced points don't cluster into lines. |

## Why sort by fp for ordering

- `store.roomsFor()` already returns rooms `ORDER BY r.fp ASC`.
- Fp is a Blake3 hash of the room's Ed25519 public key — byte-stable across
  any serialisation round-trip.
- Sorting by fp is the lightest-weight total order with no additional state.
- If room fps change (unlikely post-minting), the grid shifts — but so would
  any ordering scheme. The AC3 invariant is "same palace → same grid", which
  holds as long as the fp set is unchanged.

## The console.info call (AC3 requirement)

One `console.info` is emitted the first time any room falls back to the grid:

```
[PalaceLens] palace "<fp>": one or more rooms have no jelly.layout —
using deterministic Fibonacci-shell grid fallback (AC3).
```

The flag `gridFallbackUsed` prevents duplicate emissions on re-renders.
This matches the "single console.info on fallback" requirement in AC3.

## Shell radius choice

5 m is chosen as the default shell radius because:
- The orbit camera starts at z=12, giving a comfortable framing for a 5 m shell.
- Onion-shell sphere radius (0.35 m) fits visually at 5 m without nodes overlapping
  until N > ~120 rooms (well beyond the 500-room NFR10 upper bound, where layout
  should be explicit).
- The ADR (2026-04-24-coord-frames §2) notes radius is in canonical meters;
  5 m is a reasonable "room-scale" palace interior.

## Future

When `jelly.layout` coverage improves (S5.3+), the fallback path will be
exercised less. The algorithm is retained as a permanent fallback — a palace
with rooms freshly added to the store but not yet arranged by the curator will
always have a sensible default view.
