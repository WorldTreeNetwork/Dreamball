# S5.3 — RoomLens Interior Grid Fallback Algorithm

**Date**: 2026-04-24
**Story**: 5.3 — `RoomLens.svelte` interior layout (FR16)
**Author**: claude-sonnet-4-6 via `oh-my-claudecode:executor`

## Why planar XZ, not a shell

PalaceLens (S5.2) uses a Fibonacci-spiral shell at radius 5 m for outer-palace
room placement. RoomLens is the interior view of a single room — a fundamentally
different geometric context. Rooms are interior-scale (walls, floor, ceiling),
so inscriptions live in a roughly planar arrangement on or near the floor plane.
A shell fallback would scatter inscriptions in a sphere around the room origin,
which is both spatially wrong (inscriptions floating at ceiling height) and
visually disorienting for a reader standing inside the room. A flat planar grid
in the XZ plane at Y=0.5 (inscription mid-height) is the natural interior
analogue: inscriptions rest on the floor, evenly spaced, easy to navigate with
orbit controls. The shell pattern belongs to the exterior-palace scale; the grid
pattern belongs to the interior-room scale.

## Fp-sort ordering and byte-stability

`store.roomContents(roomFp)` returns inscriptions sorted by fp lexicographically
(`ORDER BY fp ASC`). The grid-fallback algorithm assigns positions by array index:
`index i → row = floor(i / cols), col = i % cols`. Because the fp-sort order is
deterministic and the algorithm is a pure function of index and total, the same
fp set always maps to the same grid positions across independent mounts, across
server restarts, and across browser reloads. Byte-stability is asserted by
`RoomLens.test.ts` AC2 block using `Float64Array` buffer comparison (not
floating-point approximate comparison) to rule out any nondeterminism in the
arithmetic. This property matters for the Memory Palace aesthetic: inscriptions
should feel anchored to their positions, not drift on reload.

## Default-facing = room centroid

When a `RoomContentsItem` carries no `placement.facing`, the inscription needs a
default orientation. Two candidates were considered:

1. **Camera-origin facing** — inscriptions face `[0,0,0]` in world space, which
   approximates facing the viewer's initial camera position.
2. **Room centroid facing** — inscriptions face the average position of all items
   in the room (placed or fallback).

Camera-origin facing was rejected because the camera moves — orbit controls let
the viewer circle the room, making a static "face the initial camera" orientation
wrong from any non-trivial viewpoint. A well-placed inscription would face the
viewer only from the initial angle and present its back from all others.

Room centroid is a stable geometric property of the room's contents. All
inscriptions facing inward toward the centroid creates a natural "circle of
knowledge" arrangement: every inscription is readable from the room's centre.
This is also more semantically appropriate for a Memory Palace — inscriptions
address each other, not just the viewer. The centroid is computed from all
inscription positions (placed or fallback) before the per-item quaternion
derivation loop, so the facing computation is a single-pass operation with no
circular dependency.

## GRID_SPACING choice

`GRID_SPACING = 1.5` meters. Interior rooms are modelled at roughly human scale
(a room might be 6–10 m across). At 1.5 m spacing a 50-inscription grid spans
approximately `ceil(sqrt(50)) = 8` columns × `ceil(50/8) = 7` rows = a 10.5 m ×
9 m footprint, which fills a mid-size room comfortably without items overlapping.
The inscription placeholder mesh is 0.5 m wide × 0.7 m tall, so 1.5 m inter-item
spacing provides 1.0 m of clear gap between adjacent items — readable from a
standing viewpoint. A tighter spacing (e.g. 1.0 m) was considered but risks
visual crowding at 50 items; a wider spacing (e.g. 2.0 m) pushes outer-row items
beyond typical room walls. 1.5 m is a comfortable default; the constant is named
`GRID_SPACING` in the component and can be adjusted in a single place if a future
story requires a different interior scale.
