# S5.5 — Traversal Transaction Shape

_Date: 2026-04-24_

## Why a single transaction?

`store.recordTraversal` performs five logical steps within a single async call chain that the caller awaits before painting the traversal arc. The steps are:

1. **Check aqueduct existence** — determines whether to set `aqueductCreated` in the result tuple.
2. **`getOrCreateAqueduct`** — if missing, materialises the `Aqueduct` row with D-003 defaults (`resistance: 0.3`, `capacitance: 0.5`, `strength: 0`) and emits a paired `aqueduct-created` `ActionLog` entry. Both the aqueduct fp and the action fp are derived deterministically from `(fromFp, toFp, palaceFp)` — no timestamps in the fp, so replay from `ActionLog` reproduces the same fp.
3. **`updateAqueductStrength`** — Hebbian saturating update (`s ← s + 0.1 × (1 − s)`). TC17 monotone: strength never decreases on the signed chain.
4. **`recordAction` for `move`** — emits a `move` `ActionLog` entry. The fp is `deriveTripleFp(fromFp, toFp, "move", timestamp)` — replay-stable and serves as the Blake3 handle the renderer awaits (SEC11 ordering).
5. **Read-back** — a final `MATCH` on the aqueduct row returns the post-update `strength`, `conductance`, and `revision` values. The renderer receives these only after the write chain resolves.

The atomicity is _logical_, not database-transactional — LadybugDB (`@ladybugdb/core`) uses per-statement Cypher, not multi-statement transactions. The chain is therefore "commit-ordered": each step's write is durable before the next step reads. If any step throws, the promise rejects and the renderer never paints the arc (SEC11 fail-closed).

## Dual-signature ordering (SEC11 + NFR12)

The story spec requires dual Ed25519 + ML-DSA-87 signatures on the `move` action (NFR12). The Zig-side WASM signer parameterisation over arbitrary keypairs is a known gap (`known-gaps §6 — TODO-CRYPTO`). In the MVP:

- The `move` `ActionLog` row's `cbor_bytes_blake3` field carries the derived action fp itself as a sentinel pointer.
- This makes the row well-formed and gives the renderer a stable Blake3 handle.
- When the Zig signer lands, `recordTraversal` will round-trip the action envelope bytes through `jelly.wasm`, receive the signed bytes back, Blake3 them, and store _that_ hash as `cbor_bytes_blake3`. The domain shape is identical; only the pointer value changes.

The renderer's SEC11 contract is satisfied today: it awaits the returned promise before painting, and the promise resolves only after the `ActionLog` row is persisted.

## Why `fromFp` defaults to `palaceFp` on first navigate?

`DreamBallViewer` tracks `currentRoomFp` in `$state`, starting at `null`. On the first navigate event, `currentRoomFp` is null, so `fromFp` falls back to `palaceFp`. This matches the Zig CLI convention for the first `move` action in a palace session (actor enters from the palace entrance, not from a room). The aqueduct `palaceFp → firstRoomFp` is created with D-003 defaults and represents the "entrance traversal". Subsequent navigates carry a valid `fromFp = currentRoomFp`.

## Renderer awaits Blake3 (SEC11)

The traversal arc is painted in `DreamBallViewer.svelte` inside the `onTraversal` callback, which is invoked from the `handleNavigate` function after `await store.recordTraversal(...)` resolves. The `DustCobweb` and `AqueductFlow` uniform updates therefore always happen post-persist. This is the sprint-001 _strict_ mode (per the open question in S5.5 Technical Notes: "Default strict for sprint-001; optimistic Growth").
