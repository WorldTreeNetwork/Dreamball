## Aesthetic & performance decisions (document-the-why)

### Why "subtle golden thread" is the default
Per Story 5.1 direction, the *primary* visual is a thin luminous filament
tracing the traversal path — not a literal water-carrying earthwork. The
filament design:
- Keeps 50-aqueduct rooms visually calm (busy geometry becomes noise).
- Fits the ≤2ms / 50-aqueduct frame budget trivially on an Apple M1 class
  GPU (one draw call per aqueduct, ≤48 instances each).
- Gives conductance and freshness clear single-dimensional visual coupling
  — luminance + hue — which reads well at a glance without legend.

### Why EARTHWORK is parametric, not hand-authored
The `uMode = 1` path expands the tube radius 3× and adds a second fragment
blend (earth-brown base + teal water ripple + gold thread overlay). Everything
is driven by uniforms the wrapper already provides. Hand-authored earthwork
geometry (e.g. glTF-imported channel meshes) was deferred because:
- The parametric variant is one shader switch, no asset pipeline, no binary
  size cost.
- Story 5.1 explicitly marks hand-authored earthwork as "deferred — default
  parametric".
- The variant is opt-in via a Storybook control; the showcase use case is
  celebrating a single strong aqueduct, where parametric channels already
  look distinctly different from the default thread.

### Why the `maxParticles` ceiling is 48
The wrapper clamps particle count to `clamp(round(capacity × strength × 64), 4, 48)`.
Reasoning:
- 48 × 50 aqueducts = 2,400 instanced quads → well inside a single draw
  call's budget for an Apple M1 GPU (empirically ≈10k instances before
  2ms is approached).
- `Math.round(cap × str × 64)` saturates to 64 at `cap=1.0 × str=1.0`; the
  48-ceiling applies ~25% headroom to absorb subsequent overlays (rooms
  mid-floor, avatars drawing on top) without re-tuning.
- The minimum floor of 4 particles keeps the filament visually alive on
  brand-new (strength ≈ 0) aqueducts — without it, the "just-created"
  state would render as empty.

### Why `freshness()` is exported from `aqueduct.ts`, not duplicated in the wrapper
R7 parity requirement: the renderer uniform and the (future) server-side
oracle computation must return byte-identical `Float64` values. By importing
the exact same pure function in both call sites we get bit-identity **as a
compile-time guarantee** instead of a test discipline. The new `freshness(now,
lastTraversed, tau?)` alias sits above the older `freshnessForRender(_, t, tau)`
so Epic 5 consumers get the call-site shape Story 5.1 AC (b) specifies,
without splitting the formula implementation.

### Why uniforms are pushed imperatively in a `$effect`, not supplied as reactive Svelte bindings
Three.js updates uniforms by mutating `material.uniforms[name].value` and
leaves `material.version` untouched. Recreating the `ShaderMaterial` on every
prop tick would rebuild the GL program, blowing the resize-stability contract
(AC (e)). So the wrapper constructs the material once with neutral defaults
and pushes prop-derived values inside a single reactive effect.

## Dev ergonomics

- Shader files live under `src/lib/shaders/` (path suggested by Story 5.1
  Technical Notes). Vite's native `?raw` import loads them as strings — no
  GLSL-loader plugin needed.
- The wrapper is pure Svelte 5 runes (`$props`, `$derived`, `$effect`) and
  Threlte v8's `<T.Mesh>` primitive; no component-library magic.
- A 19-test Vitest file (`AqueductFlow.test.ts`) colocated with the wrapper
  covers the six checkboxes headlessly; the Storybook play-test covers the
  live-GPU slice (pixel-delta, frame budget).

## What's NOT in this spike (scope discipline)

- **No store integration.** The wrapper accepts `AqueductUniforms` via
  props; S5.2's `PalaceLens.svelte` wires the actual store.ts-derived
  value. This is intentional — it keeps S5.1 focused on the shader
  contract and avoids cross-epic coupling.
- **No traversal event emission.** That's S5.5 (`store.recordTraversal`
  round-trip). S5.1 renders a static-ish aqueduct driven by uniforms.
- **No multi-aqueduct path routing.** The spike uses a straight segment
  from (-1, 0, 0) to (+1, 0, 0); real path geometry comes in with
  PalaceLens (rooms-connected-to-rooms). The shader interpolates along
  whatever endpoints the wrapper passes, so extending to curves later is
  additive.

## Follow-ups (out of S5.1 scope, logged for later stories)

1. **Path-endpoint uniforms.** Replace the hard-coded `(-1,0,0) → (+1,0,0)`
   segment with `uFrom`/`uTo` uniforms supplied by `PalaceLens` (S5.2).
2. **Curve primitive.** For 3-hop aqueducts, swap the linear `mix(from, to, t)`
   for a Catmull-Rom spline. Particle count ceiling stays; the shader math
   changes to a cubic.
3. **WebGPU `TSL` port.** When Three.js `WebGPURenderer` lands as non-
   experimental, port the GLSL to TSL so the same material auto-compiles to
   both backends. Current shader is simple enough that the port is ≤30 lines.
4. **Earthwork banks as actual geometry.** The parametric banks look fine in
   the spike but a hand-authored mesh (gltf) would read better for the
   showcase case. Growth tier.
